# Payments and checkout

These routes power the real payment flows in Singularity Layer.

## Endpoint direct access

### `GET /e/:slug`

Base URL:

```text
https://api.x402layer.cc/e/:slug
```

Purpose:

- return a `402 Payment Required` challenge for direct paid endpoints
- verify `X-Payment`
- proxy the request after settlement

This is the main x402 pay-per-request flow.

## Endpoint credit purchase

### `GET /e/:slug?action=purchase`

Purpose:

- request an x402 challenge for credit-pack purchase
- buy credits for later low-latency consumption

Hosted UI companion:

```text
https://studio.x402layer.cc/pay/credits/:slug
```

## Product purchase

### `GET /p/:slug`

Base URL:

```text
https://api.x402layer.cc/p/:slug
```

Purpose:

- return an x402 challenge for a product purchase
- verify and settle payment
- return file/text delivery payloads after payment

Hosted UI companion:

```text
https://studio.x402layer.cc/pay/:id-or-slug
```

## Subscription products

Subscriptions are currently modeled as canonical plan products:

- `plan-pro-monthly`
- `plan-pro-yearly`
- `plan-business-monthly`
- `plan-business-yearly`

Hosted UI route:

```text
https://studio.x402layer.cc/pay/subscription/:tier
```

Under the hood, the page fetches the x402 challenge for the relevant plan product and uses the same product-backed payment flow.

## Coupon-aware subscription checkout

Current model:

- free coupons bypass x402 entirely
- discounted coupons use a signed coupon token and reservation flow before checkout
- the worker verifies the signed token before lowering the x402 challenge amount

This keeps the worker from trusting raw client-supplied discounted amounts.

## Common payment headers and concepts

| Item | Meaning |
| --- | --- |
| `X-Payment` | x402 payment proof |
| `client_reference_id` | seller or app correlation reference |
| `metadata` | correlation metadata echoed into downstream records and receipts |

Use this surface when you need:

- credit top-ups
- subscription renewal or upgrade flows
